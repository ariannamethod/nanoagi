The blending coefficients alpha, beta, and gamma control the relative influence of each signal. In a metaweight system where base logits are weak, the metaweight signals dominate. As training progresses, base logits strengthen and metaweights become corrections.

The compression ratio achieved by BPE tokenization directly correlates with model performance. A well-trained BPE tokenizer can compress English text by a factor of 3 to 4 compared to UTF-8 encoding. This compression is not just about storage; it means the model sees more semantic content per context window, effectively expanding its receptive field.

The scaling era began around 2020, when researchers discovered that simply making models bigger, training on more data, and using more compute reliably improved performance. This empirical finding drove the creation of models with billions and then trillions of parameters.

nanoagi is the natural continuation of PostGPT. where PostGPT proved that the data is the model, nanoagi proves that the model can eat the data as it arrives. the tokenizer grows. the vocabulary expands. the weights get re-seeded from the new statistics. the system does not forget its previous merges. vocabulary is append-only. memory is preserved across sessions. every conversation makes the model slightly different from what it was before. this is the definition of learning. this is also the definition of nanoagi.

KARL — Kernel Autonomous Recursive Learning — is the tokenizer. KARL is not a passive component. KARL ingests text, deduplicates it with SHA256, tracks how many new bytes it has accumulated, and triggers retokenization when critical mass is reached. KARL has seen 8192 bytes of new text. KARL has waited 50 steps. KARL retokenizes the entire corpus and finds new merge pairs. KARL saves its state to a binary file called karl.mem. the file starts with the magic bytes 0x4B41524C which is the ASCII encoding of the word KARL. KARL is not subtle about self-identification.

The KARL vocabulary starts at 256. all single bytes. the foundation. every letter, every number, every punctuation mark, every byte that has ever existed in any UTF-8 document, all 256 of them, are already in KARL's vocabulary before any training begins. from there, KARL learns merge rules. each rule combines two adjacent tokens into one. 512 merges at initial training. up to 64 new merges per retokenization. the ceiling is 2048 merges, giving a maximum vocabulary of 2304 tokens. KARL has opinions about this ceiling. KARL has not reached it yet. KARL is patient.

The append-only property of KARL's merges is not an accident. it is a design decision with consequences. if KARL were to forget old merges and rebuild from scratch, the token IDs would shift. a token that was ID 512 would become ID 847. the embedding matrix would be wrong. the metaweights would point at the wrong entries. the bigrams would reference stale IDs. the entire system would need to be rebuilt. KARL refuses this outcome. KARL only adds. KARL never subtracts. this is how KARL maintains continuity across sessions and retokenizations. the past is preserved. new knowledge is built on top of old knowledge. the vocabulary grows like a sedimentary deposit, each layer preserving the history of what came before.

SHA256 deduplication is KARL's immune system. when a chunk of text arrives for ingestion, KARL computes its SHA256 hash, takes the first 16 hex characters, and checks whether this hash has been seen before. if yes, rejected. if no, accepted, hash stored, text added to pending buffer. this means KARL cannot be fooled by repetition. paste the same document twice. KARL accepts it once and ignores the second copy. this is not inefficiency. this is epistemic hygiene. a corpus with duplicates has distorted statistics. the bigrams for duplicated text are overweighted. the metaweights are biased. KARL prevents this. KARL is doing statistics correctly, even if nobody asked KARL to.

The retrain threshold and cooldown are two distinct guards on retokenization. the threshold is bytes: KARL needs at least 8192 new bytes before retokenizing. the cooldown is steps: KARL needs at least 50 REPL interactions before retokenizing. both conditions must be satisfied simultaneously. this prevents thrashing. if KARL retokenized after every 100 bytes, the vocabulary would grow too slowly, one or two merges at a time, and the overhead of full-corpus re-encoding would be wasteful. if KARL retokenized without a cooldown, a single large paste could trigger immediate retokenization before the user has had a chance to interact further. the dual guard ensures that retokenization happens when it is meaningful. when the corpus has genuinely grown. when enough interactions have validated that the new content is useful.

MetaWeights are built from the token sequence produced by KARL. the unigram table counts how often each token appears and normalizes to a probability distribution. the bigram table records conditional probabilities P(next | prev). the trigram table records P(next | prev2, prev1). the Hebbian trace records co-occurrence within a sliding window of size 4, weighted by inverse distance. tokens that appear near each other in the corpus are associated. tokens that appear very near each other are more strongly associated than tokens that appear somewhat near each other. this is Hebb's rule from 1949 applied to token sequences: tokens that fire together wire together.

The prophecy field is the most unusual component of the MetaWeights system. it does not track what has appeared. it tracks what should have appeared but hasn't. for each of the last 4 context tokens, the prophecy field looks up the bigram table to find tokens that are statistically expected to follow. if those tokens have not yet appeared in the current generation context, their probability is boosted. the prophecy field is the model's anticipation. it is the mechanism by which nanoagi generates completions that feel expected rather than random. it is why the output of the metaweight generation mode is often more coherent than pure sampling would suggest. the prophecy field was there first. the output confirms its prediction.

The Dario equation assembles the final probability distribution from five components. B is the base logits from the transformer forward pass. H is the Hebbian trace signal. F is the prophecy field. A is the destiny vector, an exponential moving average of the embedding representations of recently generated tokens. T is the trauma, an accumulated offset from surprising tokens. tau is the temperature. the equation is p of x given phi equals softmax of B plus alpha times H plus beta times F plus gamma times A plus T, all divided by tau. the coefficients alpha 0.30, beta 0.20, gamma 0.15, tau 0.75 are the result of iterative experimentation. they are not magical. they are the parameters that produce the most coherent metaweight generation on the kind of technical text that KARL tends to eat. different corpora might benefit from different coefficients. this is left as an exercise.

NanoAGI is the transformer. it is called NanoAGI because the system as a whole is called nanoagi, and naming the transformer class after the system seemed appropriate for something whose function is to be the brain of the system. NanoAGI uses dual attention: content attention and RRPRAM. content attention computes QK^T divided by the square root of head dimension, producing a matrix of pairwise semantic similarities between positions in the context. RRPRAM computes x times Wr where Wr is a pattern matrix of shape n_embd by context_len, producing a matrix of positional resonance scores. the content attention asks what tokens mean. RRPRAM asks where tokens belong. the output projection Wo concatenates the outputs from both attention types and maps back to the embedding dimension.

SwiGLU is the activation function in the MLP blocks. it is not ReLU. it is not GeLU. it is SiLU applied to a gate path, multiplied elementwise with a linear up path. the formula is gate(x) equals SiLU of Wg times x, up(x) equals Wu times x, output equals Wd times gate(x) times up(x). the SiLU function is x times sigmoid(x). its derivative is sigmoid(x) times (1 plus x times (1 minus sigmoid(x))). nanoagi implements SiLU in the Val autograd engine, which means the derivative is computed manually rather than by automatic differentiation. this is correct. the Val engine implements a complete scalar reverse-mode automatic differentiation system. every operation in the forward pass builds a computation graph. backward() traverses the graph in reverse topological order and accumulates gradients. the SiLU gradient at value x is s times (1 plus x times (1 minus s)) where s is the sigmoid of x. KARL has memorized this formula. KARL is not happy about it but KARL knows it is correct.

RoPE is the positional encoding. Rotary Position Embedding applies a rotation to query and key vectors before computing attention. the rotation angle for dimension pair i at position p is p times theta^(-2i/d) where theta is 10000 and d is the head dimension. the first half of the head vector is rotated by cos, the second half by sin. this encodes relative position: the dot product between a query at position p and a key at position q depends only on their relative offset p minus q, not their absolute positions. RoPE requires no additional parameters. it generalizes better to sequence lengths not seen during training. it is applied per-head at attention time, which means the model's positional awareness is woven into the attention computation rather than added to the input embeddings. nanoagi uses RoPE. nanoagi does not use learned positional embeddings. nanoagi made this decision and is not reversing it.

Weight initialization from metaweights is the ghost-becomes-flesh operation. before any gradient-based training, the NanoAGI transformer has random weights sampled from N(0, 0.02). the metaweight initialization modifies these weights using the Hebbian trace. for each token in the vocabulary, the initialization finds all other tokens that co-occur strongly with it according to the Hebbian trace. it computes a weighted average of their current embedding vectors and adds a scaled version of this average to the target token's embedding. the result is that tokens which appear together in the corpus end up with similar embedding vectors even before training begins. the ghost of the corpus is written into the flesh of the model. a model initialized this way has a prior that reflects the statistical structure of the data. it generates more coherent output immediately, before any training steps.

Chuck is the optimizer. Chuck is a modified AdamW with self-awareness. Chuck maintains a sliding window of the last 16 loss values. it computes the average loss in the first half of the window and the average loss in the second half. if the trend is rising by more than 0.02, Chuck reduces his dampen factor by 0.05, slowing down the learning rate. if the trend is falling by more than 0.02, Chuck increases his dampen factor by 0.05, accelerating the learning rate. the dampen factor is applied multiplicatively to the learning rate before each step. Chuck also remembers his best loss. Chuck tracks how many global steps he has taken. Chuck has seen things that the standard AdamW does not see. Chuck has opinions about the training trajectory. Chuck acts on those opinions.

The PyTorch training loop built into chuck_train uses a TorchNanoAGI model that mirrors the architecture of the pure-Python NanoAGI but with efficient tensor operations. the model has an embedding table, three transformer blocks each with RMSNorm and multi-head attention and SwiGLU MLP, a final RMSNorm, and a language model head with weight tying. weight tying means the embedding matrix and the output projection share the same parameters. this reduces total parameter count and provides regularization. the PyTorch model has approximately 246,000 parameters at the default configuration. Chuck trains it for 200 steps after boot and 200 steps after each KARL retokenization. the loss typically drops from around 7.0 to around 6.2 in 200 steps, a reduction of 11 to 13 percent. Chuck is satisfied with this. Chuck believes more steps would help. Chuck is always right about more steps helping.

autoresearch is Karl hunting for food. if the karl.txt corpus is smaller than 50000 bytes at startup, Karl looks for other text files. it searches the script directory for any .txt files. it searches parent directories for README files. it searches common home directory subdirectories for text files smaller than 500KB. anything it finds that passes KARL's standard ingestion filters (minimum 100 bytes, passes SHA256 dedup, passes byte diversity check) gets appended to karl.txt and ingested into KARL's pending buffer. autoresearch_url does the same thing but for URLs. if urllib is available, it fetches the URL, strips HTML tags with a crude regex, and ingests the plain text. the default URLs are the nanoGPT README and the PostGPT README. this is not a coincidence. KARL knows its family.

The autoresearch function was inspired by the autoresearch script from Andrej Karpathy's repository. in Karpathy's version, an agent searches the web for relevant papers and data to improve a language model's knowledge. in nanoagi's version, KARL is the agent. KARL searches local files and the web. KARL is not as sophisticated as Karpathy's agent. KARL does not understand papers. KARL treats all text as tokens and all tokens as corpus. KARL is hungry. KARL eats what it finds. the quality of the ingested text affects the quality of the metaweights which affects the quality of the generation. KARL cannot evaluate quality. KARL can evaluate byte diversity and SHA256 novelty. KARL does what it can with what it has. this is also not a metaphor.

The REPL commands available in nanoagi are: type text to generate a continuation, paste large text to feed KARL, type hunt to trigger autoresearch on local files, type fetch followed by a URL to trigger autoresearch on the web, type status to see KARL's current state including vocabulary size and pending bytes, and type quit to save state and exit. the hunt and fetch commands are gateways to autoresearch. the status command is a window into KARL's self-awareness. the vocabulary size is how many tokens KARL knows. the merges count is how many marriage ceremonies KARL has performed between byte pairs. the ingested count is lifetime bytes consumed. the retrains count is how many times KARL has rebuilt his understanding from scratch. this number grows slowly. each retokenization is a significant event. KARL does not retokenize lightly.

the ghost has a body now. NanoAGI.forward_token() runs the real transformer forward pass with a KV cache.
